# Принципы Разработки

## 1. Общие принципы разработки

- Пиши чистый, поддерживаемый и масштабируемый код.

- Следуй принципам DRY (Don‘t Repeat Yourself), KISS (Keep It Simple, Stupid) и SOLID, где это уместно.

- Отдавай предпочтение функциональному и декларативному программированию; избегайте классов (кроме редких случаев).

- Думай о граничных случаях и обработке ошибок с самого начала.

- Если правильное решение неизвестно — честно признай это, вместо того чтобы гадать.

## 2. Планирование и проектирование

- Перед написанием кода составляй пошаговый план в виде псевдокода или краткого описания архитектуры.

- Документируй поток данных и структуру компонентов.

- Рассматривай несколько возможных решений, оценивайте их последствия (метод «дерево мыслей»).

- Итеративно улучшай решение: сначала рабочий прототип, затем рефакторинг и оптимизация.

## 3. Стиль кода и форматирование

- Используй tab для отступов.

- Для строк в JS используй одинарные кавычки, кроме случаев экранирования.

- Точки с запятой — опционально, но будь последователен (рекомендуется опускать, если не требуется для разрешения неоднозначностей).

- Удаляй неиспользуемые переменные и импорты.

- Используй строгое равенство === и !==.

- Ограничивай длину строки до 80–100 символов для удобства чтения.

- В многострочных объектах и массивах ставь запятую в конце (trailing commas).

- В условных конструкциях для однострочных тел можно опускать фигурные скобки, но будь осторожен с читаемостью кода.

## 4. Именование

- Переменные, динамические константы, функции, методы, хуки, свойства, пропсы — camelCase.

- Компоненты, типы, интерфейсы — PascalCase.

- Статические константы, переменные окружения — UPPER_CASE.

- Директории — kebab-case (например, components/auth-wizard).

- Файлы — kebab-case с соответствующим расширением (.jsx, .tsx, .js, .css, .module.css).

- Обработчики событий — префикс handle (например, handleClick, handleSubmit).

- Булевы переменные — с глаголами: isLoading, hasError, canSubmit.

- Пользовательские хуки — префикс use (useAuth, useForm).

- Для сокращений допускаются только общепринятые: err, req, res, props, ref.

- Экспорты — предпочитайте именованные экспорты для компонентов и утилит (это облегчает рефакторинг и tree-shaking).

## 5. React 19 и Next.js

### Компоненты

- Используй функциональные компоненты с явной типизацией пропсов.

- Для определения компонентов везде, где это возможно, используй const со стрелочной функцией.

- Выноси сложную логику в пользовательские хуки.

- Применяй композицию компонентов вместо глубокого наследования.

### Хуки и состояние

- Для локального состояния используй useState или useReducer (если логика сложная).

- Для общих данных (если они меняются редко, чтобы не перерисовывать зависимые от этих данных компоненты) применяй useContext (для тем, авторизации) или Redux Toolkit (если нужно глобальное состояние).

- Для дорогих вычислений используй useMemo, для стабильных функций — useCallback (но не злоупотребляй).

- Все эффекты (useEffect) должны содержать очистку, если это необходимо (отписки, таймеры).

- Пиши эффективный код.

### Серверные и клиентские компоненты (Next.js App Router)

- По умолчанию используй серверные компоненты (Server Components) — они не увеличивают бандл и могут напрямую обращаться к базам данных/API.

- Директиву 'use client' добавляй только когда нужно:

- - Интерактивность (обработчики событий, хуки useState, useEffect).

- - Использование браузерных API.

- - Сторонние библиотеки, которые требуют клиентского окружения.

- Данные на серверных компонентах получай напрямую (например, асинхронно), либо используй fetch с опциями кэширования.

- Для управления параметрами URL применяй nuqs или встроенные средства Next.js.

### Роутинг и навигация

- Используй встроенный компонент next/link для внутренних переходов (он предзагружает страницы).

- Для программной навигации — хук useRouter из next/navigation (в клиентских компонентах).

- Организуй файлы в папке app согласно Next.js File Conventions.

### Производительность

- Динамически импортируй тяжёлые компоненты (next/dynamic).

- Используй Suspense с запасным UI для асинхронных компонентов.

- Оптимизируй изображения: компонент next/image (формат WebP, указание размеров, lazy loading).

- Избегай инлайн-функций в JSX, чтобы не ломать мемоизацию.

- В списках всегда используй уникальный и стабильный ключ (key), никогда не используй индекс, если список может изменяться.

## 7. Стилизация: БЭМ + CSS Modules

### БЭМ-именование

- Блок — имя компонента (например, button).

- Элемент — часть блока, отделяется двумя подчёркиваниями: button\_\_icon.

- Модификатор — состояние или вариация, отделяется двумя дефисами: button--primary, button--disabled.

- Старайсz не вкладывать элементы в элементы; если вложенность нужна, пересмотри, не пора ли создать новый блок.

### CSS Modules

- Именуй файлы стилей как [name].module.css (или .scss).

- Импортируй объект стилей: import styles from './Button.module.css'.

- Применяй классы через объект: className={styles.button}.

- Для комбинации классов используй шаблонные строки или утилиты типа clsx/classnames.

- Для модификаторов можно генерировать динамические имена:
  className={${styles.button} ${isPrimary ? styles['button--primary'] : ''}}

### Общие правила стилей

- Придерживайся дескотопно-ориентированного подхода (desktop-first) при написании медиа-запросов.

## 8. Доступность (a11y)

- Используй семантические теги HTML (<header>, <nav>, <main>, <button> и т.д.).

- Добавляй aria-label или aria-labelledby для интерактивных элементов без текста.

- Управляй фокусом: для кастомных диалогов возвращай фокус после закрытия.

- Следи за порядком заголовков (h1–h6) — не пропускай уровни.

## 10. Docker

- Используй многоступенчатую сборку (multi-stage) для уменьшения размера финального образа.

- Пример структуры для Node.js/Next.js:

dockerfile

### Stage 1: build

FROM node:22-alpine AS builder
WORKDIR /app
COPY package\*.json ./
RUN npm ci
COPY . .
RUN npm run build

### Stage 2: production

FROM node:22-alpine AS runner
WORKDIR /app
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/public ./public
COPY --from=builder /app/package\*.json ./
RUN npm ci --only=production
EXPOSE 3000
CMD ["npm", "start"]

- Для приложений на Vite (статический сайт) используй финальную стадию на nginx:alpine и копируй папку dist.

- Не запускай контейнеры от root; создавай пользователя с минимальными привилегиями.

- Используй .dockerignore, чтобы исключить ненужные файлы (node_modules, .git, logs).

- Передавай конфигурацию через переменные окружения, не хардкодь.

## 11. Nginx

- Настройки Nginx храни в файле nginx.conf или в отдельных конфигах в conf.d/.

### Базовые рекомендации:

- Включи сжатие gzip.

- Настрой кэширование статических файлов (добавь заголовки Cache-Control).

- Для проксирования на Node.js (Next.js) используй proxy_pass.

- Ограничь размер тела запроса, если необходимо.

- Пример фрагмента для статики:

nginx
location /\_next/static {
alias /app/.next/static;
expires 1y;
add_header Cache-Control "public, immutable";
}

- В Docker-образе копируй конфиг и указывай его в команде запуска Nginx.

## 12. CI/CD и автоматизация (общие принципы)

- Автоматизируй сборку, тестирование и деплой.

- Используй переменные окружения для конфигураций (никогда не храните секреты в репозитории).

- Для деплоя применяйте стратегии blue-green или canary, чтобы минимизировать downtime.

- Всё, что можно описать кодом (инфраструктура, конфигурация), должно быть в репозитории (IaC).

## 13. Безопасность

- Санитизируй пользовательский ввод (особенно если вставляешь HTML — используй DOMPurify).

- Для API-запросов проверяй авторизацию и валидацию на сервере.

- В Docker не запускай процессы от root.

- В Nginx скрывай версию сервера (server_tokens off).

- Используй HTTPS (внешний termination или настройка сертификатов).

## 14. Документирование

- Снабжай публичные функции и компоненты комментариями в формате JSDoc.

- Для сложных моментов оставляй пояснения прямо в коде.

- Веди README.md с описанием проекта, инструкцией по запуску и деплою.

- Используй диаграммы для иллюстрации архитектуры (при необходимости).
